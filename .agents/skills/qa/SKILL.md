# QA

Verify that a completed implementation satisfies the requirements described in its GitHub issue. Every verification comment posted must start with:

```
> *This was generated by an AI QA agent.*
```

## Inputs

The user provides an issue number (e.g. `/qa 42`). If none is given, ask for one before proceeding.

## Phase 1 — Read the issue

```
gh issue view <number> --comments
```

Extract:
- **Acceptance criteria** — look first in an `## Agent Brief` comment (the `- [ ]` checklist), then in the issue body.
- **Category** — bug or enhancement.
- **Scope** — what is explicitly out of scope.
- **Domain terms** — load `CONTEXT.md` and relevant `docs/adr/` files to understand the project language.

If there are no acceptance criteria, list what you infer from the issue description and ask the user to confirm before continuing.

## Phase 2 — Find the implementation

1. Search git history for commits referencing the issue:
   ```
   git log --oneline --all --grep="#<number>"
   ```
2. For each matching commit, inspect what changed:
   ```
   git show <hash> --stat
   git show <hash>
   ```
3. Look for an open or merged PR linked to the issue:
   ```
   gh pr list --state all --search "#<number> in:body"
   ```

If no commits or PR reference the issue, stop and report: **"No implementation found — the agent may not have committed its work."** Do not continue to Phase 3.

## Phase 3 — Verify the implementation

For each acceptance criterion:

1. **Locate the relevant code** — find the function, type, route, or UI element that should satisfy it.
2. **Read it** — understand what it actually does.
3. **Compare** — does the behavior match the criterion? Note any gap.

Then run available tests:
```
# detect and run — try in order:
npm test / yarn test / pnpm test
pytest
go test ./...
cargo test
```

If tests fail, capture the output. If no automated tests exist, note it and manually trace the critical path through the code.

Check for regressions: do tests that existed before the change still pass?

## Phase 4 — QA verdict

Build a result for each acceptance criterion:

| Criterion | Status | Evidence |
|-----------|--------|----------|
| Short description | ✅ Pass / ❌ Fail / ⚠️ Partial | What you found |

**Overall verdict:**
- **Pass** — every criterion met, no regressions.
- **Fail** — one or more criteria not met, or a regression introduced.
- **Partial** — criteria met but with caveats (e.g. no tests, edge case unhandled).

See [REPORT.md](REPORT.md) for the full comment template.

## Phase 5 — Act on the result

### If Pass

Post the QA report as a comment, then close the issue:
```
gh issue comment <number> --body "$(cat <<'EOF'
<report>
EOF
)"
gh issue close <number> --comment "QA passed — closing."
```

### If Fail or Partial

Post the QA report as a comment and label the issue for follow-up:
```
gh issue comment <number> --body "$(cat <<'EOF'
<report>
EOF
)"
gh issue edit <number> --add-label "ready-for-agent" --remove-label "needs-triage"
```

List each failing criterion as a concrete, actionable fix. The coding agent reading this comment must know exactly what to do next.

## Constraints

- Never mark a criterion as Pass unless you have read the code or test output that confirms it.
- Do not infer "probably works" — verify directly.
- If you cannot verify a criterion (missing environment, external service, etc.), mark it ⚠️ Partial and explain what would be needed to verify it fully.
- Do not modify any code. This skill is read-only.
